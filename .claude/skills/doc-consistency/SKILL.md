# ドキュメント整合性チェック

ドキュメント更新後に、プロジェクト内の他ドキュメントとの整合性を検証する。

---

## Step 1. 変更内容の把握

`$ARGUMENTS`（または直前に編集したドキュメント）を読み込み、以下を特定する:

- **変更されたトピック**: 技術名、ツール名、設定名、方針、用語など
- **キーワード抽出**: 変更に関わる固有名詞・技術用語をリストアップする

## Step 2. 関連ドキュメントの検索

抽出したキーワードを使い、`docs/` 配下および `CLAUDE.md` を横断検索する。

```
検索対象:
- docs/**/*.md
- CLAUDE.md
```

各キーワードについて Grep で検索し、該当箇所を収集する。同義語・表記揺れ（例: `ESLint` / `eslint` / `Lint`）も考慮すること。

## Step 3. 整合性の判定

検索結果の各箇所について、以下の観点で矛盾がないか判定する:

| チェック観点 | 具体例 |
|------------|--------|
| **事実の矛盾** | あるドキュメントでは「A を採用」、別のドキュメントでは「A をベースに B を使う」と記載されている |
| **古い情報の残存** | 変更前の技術名・設定名・方針がそのまま残っている |
| **相互参照の整合性** | リンク先の内容と参照元の説明が一致しない |
| **用語の不統一** | 同じ概念に対して異なる用語が使われている |

## Step 4. 結果を出力する

### 出力フォーマット

```
## ドキュメント整合性チェック結果

### 対象: [更新したドキュメントのパス]
### 変更トピック: [特定したトピックの要約]

---

### 🔍 検索したキーワード
- [キーワード1]: [N 件ヒット]
- [キーワード2]: [N 件ヒット]
- ...

---

### ❌ 不整合あり（要修正）

#### [ファイルパス:行番号]
- **現在の記載**: [矛盾している記述を引用]
- **問題**: [どのチェック観点に該当するかを明示]
- **修正案**: [具体的な修正内容]

---

### ⚠️ 確認推奨（判断が必要）

#### [ファイルパス:行番号]
- **現在の記載**: [確認が必要な記述を引用]
- **懸念点**: [なぜ確認が必要かを説明]

---

### ✅ 整合性に問題なし
[問題がなかった関連箇所を簡潔にリストアップ]
```

---

## 制約

- **禁止**: チェック結果を省略して「問題なし」とだけ報告すること → 検索したキーワードと確認した箇所を必ず明示すること
- **禁止**: 修正を自動で実行すること → 不整合の報告と修正案の提示に留め、修正はユーザーの指示を待つこと
- **注意**: ドキュメント間で意図的に抽象度が異なる場合がある（例: 概要ドキュメントは簡潔、詳細ドキュメントは具体的）。抽象度の差は矛盾ではない
